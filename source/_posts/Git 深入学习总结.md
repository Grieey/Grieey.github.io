# 背景

版本控制系统`VCS(Version Control System)`，是一种用于管理控制代码版本的工具系统，目前国内主流的主要是以中央式版本系统为代表的 `SVN`和以分布式版本系统为代表的 `Git`。

- 中央式版本控制系统：顾名思义，这种类型的系统以一台计算机作为中央系统进行运行，其他的协作人员每次从中央系统中拉取和提交最新的代码。这样多个协作人员就围绕这个中央系统进行开发工作，本地不需要保存每次版本的代码。
- 分布式版本控制系统：和中央式系统最明显的区别就是，在分布式系统中，每一台计算机都作为一个控制系统存在，然后将其中一台作为主要的系统，用于各个人员协作开发。这样的系统在每一个计算机上都存在完整的代码提交记录，与中央式相比，好处就是当远程的仓库出现问题时，并不会影响其余的本地仓库进行开发。

# 基本概念

使用命令`git init`可以将一个空白的文件目录初始化为一个仓库，并且 `git`会生成一个`.git`的文件夹，这个文件夹中就会保存该目录中所有文件的改动及相应的提交记录，对应的这个目录就被称为`工作区`。在`git`中，主要由`工作区、暂存区和仓库`三部分组成。

一个基本的工作流程就是，当我们在工作区域中修改了某个文件的文件内容时，`git`会自动检测到这种改动并进行了标记，然后需要我们手动的将这些改动添加(使用命令 `add` )到暂存区中，这样我们所修改的东西就会被`git`记录下来，而没有添加到暂存区的改动在进行各种`git`命令时可能会丢失。如果我们确定了暂存区中的内容是这样的修改，就可以将暂存区的改动记录进行提交，那么这次修改的内容就会从暂存区迁移到仓库中，并在仓库中生成一个提交记录--`commit`。整个仓库中，所有的改动记录就是有一个一个的`commit`串行所构成的。

在上诉的基本工作流程的背景下，就衍生出来 `git` 中的几个重要的概念：

- 引用：引用指的是指向某个 `commit`的快捷方式，它可以是一串`SHA-1`字符串，也可以是一个`branch`分支的名字。在`git`中，每一个`commit`使用它的`SHA-1`值(一种算法计算出来的值，两个 `commit` 计算出来的值很少能重复)来作为唯一的标识，虽然可以使用该值的前几位来代表指向的是哪一个`commit`，但是这样还是很难记忆。所以在`git`中，还使用了一些特殊的字符串来表示一些引用。
- HEAD：指向当前`commit`的引用，当我们从远端仓库拉取代码时或者`checkout`新的分支时，`HEAD`指向会随着我们的操作相应的修改，使得它始终指向的是当前工作区域中对应的`commit`
- branch：分支，对应着一类引用，也就是一类`commit` 链。实际开发时，由于并行工作等因素，我们很少使用一个分支进行开发，常常是多个分支的功能同时进行着开发。因此，在`git`的仓库中，实际是由不同的分支所组成，而一个分支，则对应着一条`commit`链。在 `git`初始化时，会默认生成一个叫做`master`的分支，所以大多数工程都将该分支作为主分支使用，在开发时，建立其他分支来开发工作，最后将完善的功能合并到主分支中。

# 基本命令

### checkout

这个命令翻译为 签出，使用该命令签出某个`commit`时，工作区的内容会替换为该`commit`，并同时将 HEAD 引用指向该`commit`

### clone

大多数时候，我们都是将远程仓库克隆到本地，使用`git clone xxx(远程仓库的地址)`就可以将远程仓库克隆下来，那么这个命令具体进行了哪些操作呢。

1.首先是获取到了远程分支的镜像，将远程分支的镜像复制到本地。

2.然后根据镜像中的引用，拉取到远程的分支情况及 HEAD 引用的指向。

3.再根据`commit`的提交记录，一个一个的将远程仓库中的`commit`拉取到本地，形成本地仓库的 `commit`链

### add

我们做出的修改，需要添加到暂存区中，就使用此命令，需要注意的一点是，`git`中所记录的是文件内容的改动，而非文件本身，所以当添加了一次文件的修改后，又修改了相同文件的内容，还需要再添加一次刚刚的修改。

### commit

提交，该命令会将暂存区的改动记录提交到仓库对应的分支上

### merge

多数情况下，我们需要将不同的分支的代码进行合并，那么就需要使用到`git merge`命令，该命令具体做的事情是：**从目标 `commit` 和当前 `commit` （即 `HEAD` 所指向的 `commit`）分叉的位置起，把目标 `commit` 的路径上的所有 `commit` 的内容一并应用到当前 `commit`，然后自动生成一个新的 `commit`。**

### push

`git push ` 会将默认分支的本地提交记录上传到远程分支上进行合并，如果不指定的话，所更新的分支为`git config 中的 push.default的值对应的分支，这个值默认为:current `  其中的值`git config` 命令来进行修改， 进而改变 push 时的行为，详情查看[git config](https://git-scm.com/docs/git-config#git-config-pushdefault)。如果需要提交记录的分支不是默认的分支，需要在命令中添加几个新的参数

```shell
git push origin target_feature
```

那么这次的 push 会推向远程分支的`target_feature`分支

注意：push 时不会上传 HEAD 的指向，远程分支的 HEAD 永远指向的是 `master` 

 ### pull

拉取远程分支的提交记录到本地，并与本地分支进行合并

# 进阶处理

### merge 时几种情况的处理

1.需要 merge 的分支的提交要比当前的分支的提交多，这个时候就直接将 HEAD 指向最新的`commit`，这种在`git`中称为`fast-forward`

2.如果 merge 的分支的提交比当前的分支的少，则不会进行任何操作

3.merge时，同时目标分支和当前分支同时修改了同一文件的相同地方。这个时候，`git`就无法自动的对这类的冲突进行处理，需要我们手动的合并，不过`git`会做一些标识的工作，将两个分支不同的代码分别标记起来，让我们自己处理应该怎么合并，合并后再手动提交一次，表示冲突已经处理完成。

4.merge时，产生了冲突并不想进行这次合并了，需要使用以下命令

```shell
git merge --abort	
```

这行代码之后，仓库会回到 merge 前的状态
